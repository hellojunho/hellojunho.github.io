# SVN이란?
**SVN(SubVersion)** 이란, `Git/GitHub`와 같은 **형상관리/소스관리 도구**이다.  
SVN은 다수가 작업하는 프로젝트의 경우 버전관리나 개인이 수정한 소스의 통합과 같은 문제를 해결하기 위해 사용한다.  
> 형상관리: 소스의 변화를 끊임없이 관리하는 것
<br>

프로젝트의 소스는 SVN 서버의 **Trunk** 라는 곳에 위치하는데, 로컬 환경에 Trunk의 소스를 다운(Update) 받아 수정 및 추가 후 업로드(Commit) 하는 방식이다.  
<br>

자신만의 소스를 다른 개발자들과 독립적으로 작업하려면 **Branch(원본 소스의 나뭇가지)**를 만들어 작업한 다음, **Merge**를 하면 Trunk 소스와 병합되는 방식이다.  
- Git/GitHub와 유사하다.  

## 용어 정리
**Repository**: 프로젝트 파일 및 변경 정보가 저장되는 장소  
**Checkout**: 저장소에서 최신 버전의 소스코드를 최초로 받아오는 것  
**Import**: Repository에 맨 처음 파일들을 채우는 것  
**Export**: 버전 관리 파일들을 뺀 순수 파일만 빼내는 것  
**Update**: 로컬 저장소에 있는 파일들을 저장소의 최신 버전으로 내려 받기  
**Commit**: 로컬 저장소의 변경된 내용을 서버로 전송 / Checkout한 파일의 수정사항을 갱신함  
**Revert**: 로컬 저장소의 내용을 이전 상태로 돌림  
**Add**: 버전관리 대상으로 파일 등록  
**Trunk**: 개발 소스를 commit했을 때 개발 소스가 모여지는 곳 / 프로젝트에서 가장 중심이 되는 메인 디렉토리(소스와 파일 포함)  
**Branch**: Trunk에서 분리/복사한 소스로, 버전 별 배포판을 만들거나 Trunk와 별도로 운영환경을 위한 안정화된 소스 관리 목적  
**Tag**: 특정 시점의 상태 보존 목적으로 사용함 / 장기적으로 1.0, 1.1 등의 버전 별로 소스코드를 따로 저장  

## 장점
- `원자적 커밋`을 지원하므로 다른 사용자의 커밋과 엉키지 않음 -> 실패 시 롤백 가능  
- 직관적  
파일과 디렉토리의 삭제, 이동, 이름 변경, 복사 지원  
- 소스파일 이외의 `바이너리 파일`도 효율적으로 저장 가능  
- 디렉터리도 버전 관리 가능  
- 저장소의 크기에 상관 없이 일정한 시간 안에 가지치기(branch)나 태그 가능  
- 처리 속도가 상대적으로 빠름

## 단점
- 소스코드는 Merge(병합)가 가능하지만, 바이너리 파일은 어느 한 쪽을 버릴 수 밖에 없음  
- 개별 개발자만의 개발 이력을 가질 수 없음  
- `.svn` 디렉터리로 인해 저장소가 다소 지저분해보임  
- 잦은 커밋으로 인해 리비전 번호가 크게 증가할 수 있음  
- 충돌 확률이 높음  

## Git(GitHub)와의 차이점
### 기본 구조 차이
**SVN**  
SVN은 중앙 서버에서 전체 코드 변경 및 이력을 저장하는 **중앙집중식 구조**를 사용한다.  
개발자들은 중앙 서버와 연결해야 하며, 작업한 내용을 중앙 서버에 커밋하고 최신 변경 사항을 받아와야한다.  
<br>

**Git**  
Git은 **분산 구조**를 사용하며, 각 개발자의 로컬 저장소에 전체 코드 및 변경 이력이 `복제`된다.  
개발자들은 로컬 저장소에서 작업하고, 변경 사항을 원격 저장소와 동기화 한다.  
이로 인해 오프라인에서도 작업이 가능하다는 상대적 이점이 있다.  

### 워크플로우
**SVN**  
SVN은 작업 시작 전에 중앙 서버에서 최신 코드를 받아온 후(Update), 작업이 끝난 후 중앙 서버에 변경 사항을 반영(Commit)한다.  
변경 사항이 겹치는 경우 충돌이 발생할 수 있고, 충돌이 발생하면 해결을 해야한 커밋이 가능하다.  
<br>

**Git**  
Git은 개발자가 로컬 저장소에서 독립적으로 작업할 수 있기 때문에, 작업 후 로컬 저장소에 커밋하고 원격 저장소와 동기화 할 때만 인터넷 연결이 필요하다.  
충돌이 발생할 경우 로컬에서 충돌을 해결하고, 원격 저장소에 반영(Push)한다.  

### 브랜치 관리
**SVN**  
SVN은 브랜치를 만들 때 전체 파일을 `복사`하여 새로운 디렉토리를 생성하는 방식을 사용한다.  
브랜치를 생성할 때마다 새로운 파일이 추가되고, 이로 인해 저장소의 용량이 증가한다.  
<br>

**Git**  
Git은 브랜치를 만들 때 기존의 커밋 히스토리를 참조하는 방식을 사용한다.  
따라서 브랜치를 생성해도 새로운 파일이 추가되는 것이 아니므로, 저장소의 용량이 증가하지 않는다.  
<br>

이 외에도 다양한 차이점이 존재한다.  
실제 현업에서는 두 도구 다 많이 사용되고 있다.
